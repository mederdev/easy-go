# ТЗ — Приложение водителя (режим в клиентском app)

> Статус: **черновик на согласование**. Кода нет. Реализация — после утверждения
> этого ТЗ и отрисовки макетов в Claude Design (design-first, как с кабинетом).
> Контекст: EasyGo — пассажирские трансферы (Бишкек/Алматы/Иссык-Куль).

## 1. Цель

Дать водителю собственный режим в существующем приложении `apps/client`:
вход по номеру, список своих рейсов на сегодня/ближайшие, **манифест пассажиров**
по рейсу, отметка статуса рейса (выехал/прибыл) и передача геопозиции авто.
Бэкенд уже хранит нужные данные (`Driver`, `Car`, `Flight ↔ Booking ↔ Client`) —
не хватает водительской авторизации и водительских эндпоинтов.

## 2. Базовые решения (уже согласованы)

- **Режим внутри клиентского приложения** (не отдельное приложение). После
  водительского входа интерфейс переключается на водительские экраны.
- **Вход — телефон + OTP** (как у пассажира). Пока dev-заглушка кода, реальный
  SMS/WhatsApp-провайдер — позже.
- **Водитель заводится только админом** (сущность `Driver` создаётся в CRM).
  Самостоятельной регистрации водителя нет: если телефон не привязан к водителю —
  вход запрещён.

## 3. Роли и доступ

- Новый тип JWT: `kind: 'driver'` (сейчас есть `user` для бэк-офиса и `client`
  для пассажира — см. `packages/shared/src/schemas/auth.ts`,
  `apps/api/src/plugins/auth.ts`).
- Водительские эндпоинты скоупятся по `driverId` из токена (декоратор
  `authenticateDriver`, по аналогии с `authenticateClient`).
- **Пассажирская и водительская сессии раздельны** (один активный токен в
  приложении). Предлагается: вход для водителей — отдельная точка входа
  (см. §6), переключение режима по `kind` токена.

## 4. Пользовательские сценарии

1. **Вход:** водитель открывает «Вход для водителей» → вводит телефон → код →
   попадает в водительский режим. Если телефон не зарегистрирован как водитель —
   понятная ошибка «Номер не привязан к водителю, обратитесь к оператору».
2. **Мои рейсы:** видит список рейсов, назначенных на его авто, на сегодня и
   ближайшие дни (маршрут, время, загрузка `seatsTaken/seatsTotal`, статус).
3. **Рейс/манифест:** открывает рейс → список пассажиров (имя, телефон/WhatsApp,
   кол-во мест, комментарий, адрес подачи). Может позвонить/написать пассажиру.
4. **Действия по рейсу:** «Выехал» → рейс `DEPARTED` (авто → `ON_TRIP`);
   «Прибыл/Завершить» → рейс завершён. (Опционально: отметка посадки по каждому
   пассажиру — см. открытые вопросы.)
5. **Геопозиция:** передаёт текущие координаты авто (для функции «как доехать до
   авто» у пассажира и карты у диспетчера).
6. **Профиль/выход:** видит свои данные и авто, выходит из режима водителя.

## 5. Бэкенд

Модульный монолит Fastify, паттерн `routes.ts → service.ts → Prisma`
(как в остальных модулях). Деньги — в минорных единицах. Идемпотентность —
по существующему плагину.

### 5.1. Изменения модели данных (нужна Prisma-миграция)

- **`FlightStatus`**: добавить терминальный статус прибытия. Сейчас:
  `SCHEDULED | CLOSED | DEPARTED | CANCELLED`. Предлагается добавить **`ARRIVED`**
  (рейс доехал/завершён). → миграция enum.
- **`Booking.boardedAt DateTime?`** — *опционально*, только если делаем чек-лист
  посадки в v1 (см. открытые вопросы).
- Переиспользуем существующее: `Car.currentLat/currentLng/locationCity`,
  `Car.status` (`AVAILABLE | ON_TRIP | MAINTENANCE`), `Car.driverId`,
  `Flight.carId/seatsTaken/status`, связь `Flight → Booking → Client`.

### 5.2. Эндпоинты

**Водительская авторизация** (без токена):
| Метод | Путь | Назначение |
|---|---|---|
| POST | `/driver-auth/request-otp` | `{phone}` → выдать OTP (dev: код в ответе), троттлинг как у клиента |
| POST | `/driver-auth/verify` | `{phone, code}` → проверить код **и наличие `Driver` с этим телефоном**; иначе 403. Вернуть `{token, driver}` |

**Водительские (требуют `kind:'driver'`)** — префикс `/driver`:
| Метод | Путь | Назначение |
|---|---|---|
| GET | `/driver/me` | Профиль водителя + его авто |
| GET | `/driver/trips?scope=today\|upcoming` | Рейсы, назначенные на авто водителя (route, время, загрузка, статус) |
| GET | `/driver/trips/:flightId` | Детали рейса + **манифест** (брони с клиентом: имя, телефон, pax, комментарий, подача) |
| PATCH | `/driver/trips/:flightId/status` | `{status}` — разрешённые переходы для водителя: `DEPARTED`, `ARRIVED` |
| PATCH | `/driver/trips/:flightId/passengers/:bookingId/board` | *(опц.)* отметить посадку пассажира |
| PATCH | `/driver/location` | `{lat, lng}` — обновить координаты **своего авто** |

Связь с пассажиром (звонок/WhatsApp) — на клиенте через `tel:` / `wa.me`,
отдельный эндпоинт не нужен.

### 5.3. Бизнес-правила (на согласование)

- Водитель видит и меняет **только свои** рейсы (по `car.driverId == driverId`).
  Манифест — только брони своего рейса. Чужое → 404.
- Разрешённые переходы статуса рейса водителем: `SCHEDULED/CLOSED → DEPARTED → ARRIVED`.
  Отмена рейса — только у бэк-офиса (не у водителя).
- При `DEPARTED`: авто → `ON_TRIP`. При `ARRIVED`: авто → `AVAILABLE`
  (+ обновить `locationCity` на город прибытия), `Car.tripsMonth += 1`.
- **При `ARRIVED`**: переводить ли активные брони рейса (`NEW/CONFIRMED`) в
  `COMPLETED`? → влияет на пассажирский кабинет (поездки уходят в «Историю»).
  *Предлагается: да, авто-`COMPLETED` для не-отменённых.* (открытый вопрос)
- `/driver/location` обновляет `Car.currentLat/Lng` авто водителя
  (того, что назначено на текущий активный рейс, либо единственного авто).

### 5.4. Изменения в существующем коде

- `packages/shared`: расширить `JwtClaims` ещё одним вариантом
  `DriverJwtClaims (kind:'driver', sub: driverId, name)`; добавить схемы
  driver-auth и driver-trip/manifest.
- `apps/api/src/plugins/auth.ts`: декоратор `authenticateDriver`.
- `apps/api/src/lib/otp.ts`: переиспользуется как есть (ключ — телефон).
- Новые модули `apps/api/src/modules/driver-auth/*` и `.../driver/*`,
  регистрация в `app.ts`.
- `packages/api-client`: группы `driverAuth.*` и `driver.*`.

## 6. Фронтенд (apps/client)

- **Точка входа для водителя:** ссылка «Вход для водителей» (предлагается — в
  разделе «Инфо» и/или на экране входа). Открывает водительский логин (телефон → OTP).
- **Водительский auth-store** (токен `kind:driver`, профиль); приложение по `kind`
  токена показывает **водительский режим** вместо пассажирских вкладок.
- **Водительские экраны:**
  - «Рейсы» — список на сегодня/ближайшие (карточка: маршрут, время, загрузка, статус).
  - «Рейс» — манифест пассажиров + кнопки «Выехал»/«Прибыл», связь с пассажиром,
    кнопка «Передать геопозицию».
  - «Профиль» — данные водителя, авто, «Выйти».
- **Геолокация:** плагин `@capacitor/geolocation` (добавить в зависимости) для
  пинга координат. На вебе — fallback на `navigator.geolocation`.
- Дизайн всех экранов — **сначала макеты в Claude Design**, затем верстка
  в стиле EasyGo (Manrope, зелёный `#56A919`).

## 7. Безопасность / изоляция

- Все `/driver/*` — строго по `driverId` из токена; кросс-доступ к чужим
  рейсам/манифестам запрещён (404).
- OTP: троттлинг отправки и лимит попыток (как у клиента).
- Контакты пассажиров (телефон) водитель видит только в манифесте своего рейса.

## 8. Вне scope (v2)

- Реальный SMS/WhatsApp провайдер OTP.
- Фоновое отслеживание геопозиции (background location).
- Push-уведомления водителю о новых бронях/изменениях.
- Чат водитель↔оператор; отчёты/заработок водителя.
- Возможность одновременно быть и пассажиром, и водителем в одной сессии.

## 9. Открытые вопросы (на согласование с коллегой)

1. **Один телефон = пассажир и водитель?** Сессии раздельные (предлагается) или
   одновременный доступ к обоим режимам?
2. **Жизненный цикл рейса:** хватает ли `Выехал → Прибыл`, или нужны ещё статусы
   (например, «На месте подачи»)?
3. **Чек-лист посадки пассажиров** в v1 (отметка boarded по каждому) — нужен или в v2?
4. **Авто-завершение броней** при прибытии рейса (перевод в `COMPLETED`) — ок?
5. **Геопозиция:** ручной пинг по кнопке / по таймеру в активном рейсе
   (предлагается для v1) — или ждать фоновое отслеживание (v2)?
6. **История рейсов** водителю нужна, или только сегодня/предстоящие?
7. **Точка входа водителя:** где разместить «Вход для водителей» — в «Инфо»,
   на экране входа, или скрытый вход?
8. **Несколько авто у одного водителя** — поддерживаем (модель допускает) или
   считаем «один водитель = одно активное авто»?

## 10. Критерии приёмки (как проверим)

- Бэкенд (curl, как с кабинетом): driver request-otp → verify (только для
  существующего водителя; чужой телефон → 403) → токен → `/driver/me`,
  `/driver/trips`, `/driver/trips/:id` (манифест), смена статуса (`DEPARTED`/`ARRIVED`
  с побочными эффектами на авто/брони), `/driver/location`. Изоляция: чужой рейс → 404.
- Фронт: `typecheck` + `build` чисто; вход водителя → список рейсов → манифест →
  действия. (Браузерный клик-через — при наличии разрешения на навигацию.)
